# MyFitnessPal CLI — Phase 3 Build Log

## Built

### Foundation (Priority 0)
- **Permissions hardening (commitment C-1)** — `internal/store/perms.go` plus an edit to `OpenWithContext` so the `~/.local/share/myfitnesspal-pp-cli/` directory is `0o700` and `data.db` is `0o600`. Idempotent.
- **MFP HTTP transport** — `internal/client/mfp_transport.go` rewrites the host on `/v2/*` paths from www → api, injects `mfp-client-id: mfp-main-js` everywhere, plus `Origin`/`Referer`/`mfp-user-id` on api.* requests. The numeric `mfp-user-id` is cached at `~/.config/myfitnesspal-pp-cli/mfp-state.json` (mode 0600). Wired into `newHTTPClient` so every Client benefits.
- **Diary schema add-on** — `internal/store/diary_schema.go` adds the per-food typed table `diary_entry`, day-meta table `diary_day_meta`, and FTS5 indexes `diary_entries_fts` + `foods_fts`. Side-loaded via `Store.EnsureDiaryEntries(ctx)` so it doesn't fight the generator's migration slice.

### Absorbed (Priority 1)
Generated by the press, working live as JSON:
- `food details` — `GET /v2/foods/{id}` with `--fields` selection.
- `food suggested-servings` — `GET /v2/foods/{id}/suggested_servings`.
- `food search` — POST endpoint stub (request shape only; HTML response parser deferred to v0.2).
- `measurement types` — `GET /api/user-measurements/measurements/types`.
- `water get` / `note get` — JSON islands.
- `reports get` — `GET /api/services/reports/results/{category}/{report_name}/{days}`.
- `api-user get` — `GET /v2/users/{user_id}`.
- `user auth-token` / `user top-foods-server` — JSON.

Hand-written:
- **`diary get-day`** — replaces the generator's emit. Fetches `/food/diary` with cookie auth, runs response through `internal/parser/diary.go` (ported from `python-myfitnesspal` v2.0.4), emits a structured `Day` JSON.

### Transcendence (Priority 2)
Five novel features, all hand-written and shipping:
- **`pull-diary --from --to`** — date-range sync. Iterates each day, fetches `/food/diary` with 1 req/sec pacing, parses, upserts per-food rows into `diary_entry`.
- **`export csv --from --to [--meal] [--out]`** — per-food CSV from local store. The headline thing premium MFP itself doesn't ship.
- **`find --food <q>`** — FTS5 query over `diary_entries_fts`.
- **`context --days N`** — composes daily totals + Pareto top foods + recent entries + latest goal snapshot into one JSON payload.
- **`analytics top-foods --nutrient X --days N`** — Pareto query over `diary_entry`.
- **`analytics streak --tolerance T`** — longest run inside ±tolerance of the synced calorie goal.

## Intentionally Deferred (documented as Known Gaps in README)

Seven novel features and four HTML parsers were not built in this run; they're listed under `## Known Gaps` in the printed README and are first-pass candidates for `/printing-press-polish`:

- HTML parsers: `/exercise/diary`, `/measurements/edit`, `/account/my-goals`, `POST /food/search`, `POST /food/load_recent`. The corresponding commands hit the live endpoints but emit raw HTML rather than parsed structures.
- Transcendence: `analytics weight-trend`, `analytics macro-trend`, `analytics weekly-diff`, `analytics gap-candidates`, `sync diff`, `reports backfill`, `meal expand`.

## Generator Limitations Observed

- The generator's `diary get-day` and `food search` commands assume JSON responses; for HTML endpoints the generated `resolveRead()` flow doesn't apply. Hand-written replacement of the entire generated file was the cleanest path. Filing as retro candidate: the generator could detect `response.format: html` in the spec and emit a different RunE shape that hands the body to a parser hook.
- Spec field `Param.Aliases` rejects `fields[]` even when the wire format actually requires bracket notation. Worked around by leaving the param `name` as `fields` and noting the wire format in the description; the generator's URL builder handles the encoding when the array path triggers.
- The categorical install path for the SKILL.md "Install" section is hard-coded to `library/other/...` regardless of the spec's `category` field. Fixed by hand-editing SKILL.md to match. Retro candidate: thread `category` through the canonical install template, or remove the canonical-section check on that path segment.

## Skipped Body Fields

None of the Priority 1 commands had complex request bodies that were dropped. The `food search` endpoint's POST body uses an `authenticity_token` (Rails CSRF) that the printed CLI would need to scrape from a fresh `/food/diary` GET before each call — not implemented in this build, surfaces as a Known Gap.
